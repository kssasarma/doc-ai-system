## System snapshot (what exists today)

- **`documentation-bot/`** — Spring Boot 4 / Java 21, port 8082. Chat + RAG answering, auth (JWT + refresh tokens + API keys + OIDC JIT), 30 REST controllers, multi-tenant row-level isolation (`TenantContext` ThreadLocal, fails closed), per-document + per-group ACL (`DocumentAccessPolicy` → `SearchScope`), hybrid dense+lexical retrieval with RRF fusion + MMR re-ranking, Flyway V1–V18.
- **`document-ingestor/`** — Spring Boot 4 / Java 21, port 8081. Upload → parse (Tika, structure-preserving) → chunk (`SemanticChunker`) → PII scan → embed (OpenAI) → pgvector. Storage is S3/MinIO only (no local disk). Flyway V1–V9.
- **`frontend/`** — React 18 + Vite + TS + Tailwind v3 + framer-motion + Headless UI. Design-system primitives in `src/components/ui/`, dark mode, command palette, toasts. Data fetching is raw `fetch`/axios + `useEffect` (no query library).
- **Shared Postgres 16 + pgvector**; MinIO in dev compose; Redis optional (bot only). Roles: `SUPER_ADMIN` (creates tenants only), `ADMIN` (tenant admin), `USER`. Invite-only provisioning; multi-tenant membership with active-workspace switching.
- Integrations: `slack-bot/`, `teams-bot/`, `vscode-extension/`, `browser-extension/` — all correctly use `POST /api/v1/query` with `X-API-Key`; tenant is resolved server-side from the key. Deploy: `docker-compose(.prod).yml`, `k8s/`, `helm/doc-ai-system/`, CI in `.gitlab-ci.yml`.

## The 8 product requirements this roadmap serves

1. Best-in-class, appealing frontend everyone loves to use.
2. Exceptionally solid backend; multitenancy without any issues.
3. ADMINs can configure LLMs **and API keys**.
4. SUPER_ADMIN only creates tenants. *(Already correctly enforced — `TenantController` CRUD is `SUPER_ADMIN`-only; per-tenant config is self-service. No work needed.)*
5. Each tenant admin manages their LLM models, API key, documents, and ACLs.
6. Fail-proof and consistent.
7. Cache invalidation on new document versions; product/version listings auto-update.
8. "Google for the company" — research/find anything in documents via NLP.

## Phase 2 — Document versioning, supersession, ingestion integrity & invalidation  `[CRITICAL/HIGH — requirements 6 & 7]`

**Goal:** A new version of a document cleanly replaces same-version content, search never sees failed/partial/stale chunks, downstream consumers (subscriptions, frontend product lists) learn about new versions automatically, and nothing gets permanently stuck.

- [x] **2.1 Same product+version re-upload must supersede, not duplicate.**
  - Done: `IngestionService.completeIngestion` (the new atomic-commit method from 2.3) calls `retireSupersededDocuments` after marking the new document `COMPLETED` — deletes every *other* `COMPLETED` document row (+ its chunks via `chunkRepository.deleteByDocumentId`) sharing the same `(tenantId, product, version)`. New repository method: `DocumentRepository.findByTenantIdAndProductAndVersionAndStatusAndIdNot`. No storage cleanup needed for the retired rows — a `COMPLETED` document already has `storageKey=null` (cleared at its own completion time), so there's nothing left in S3/MinIO to delete.
  - **Deliberately not done**: migrating `DocumentAccess`/`GroupDocumentAccess` grants from the superseded document to its replacement — that needs the cross-service ingestion-completed signal (2.4, not built this pass; see below) so the bot can react. Documented as a known gap directly in `completeIngestion`'s javadoc rather than silently dropped: a superseded document's grants are orphaned (inert, not reassigned), and whoever had access must be re-granted on the new upload.
  - Verified: `mvn test` — 36/36 ingestor tests pass.

- [x] **2.2 Search must only see `COMPLETED` documents.**
  - Done: added `AND d.status = 'COMPLETED'` to both `findTopNDenseAccessible`/`findTopNLexicalAccessible` (`DocumentChunkRepository.java`) and to `DocumentRepository.findDistinctProductVersionsAccessible`. Real gap found while doing this: the bot's own read-only `Document` entity (`documentation-bot/.../domain/entity/Document.java`) had **no `status` field mapped at all** — added it (plain `String`, mirroring the ingestor-owned `IngestionStatus` enum by value; the bot only ever reads this shared table, so it doesn't need the enum type). Fixed two Testcontainers test fixtures (`GroupDocumentAccessServiceTest`, `DocumentAccessIsolationTest`) that built `Document.builder()...build()` without a status, which would now fail the new `NOT NULL`-equivalent expectation for any test asserting search visibility.
  - Verified: `mvn test` — 122/122 bot tests pass.

- [x] **2.3 Make ingestion atomic per document — no committed orphan chunks.**
  - Done: `IngestionService` split into `ingestUploadedFile` (now plain `@Async`, no `@Transactional` — orchestrates only), `parseChunkAndEmbed` (parse/chunk/PII-scan/embed, no document/chunk writes, throws on any failure), and two short, separate `@Transactional` methods: `completeIngestion` (insert all chunks + flip `COMPLETED` in one commit — the only place chunks are ever persisted) and `markFailed` (used by every failure path: missing storage, parse/embed exception, and the new 2.5 reaper). `SemanticChunk` gained an `embedding` field so embeddings computed in the non-transactional phase can travel into the transactional persist phase without a parallel list. This also fixes the connection-hogging problem the same defect note flagged (a multi-minute embed loop no longer holds a DB connection).
  - Verified: `mvn test` — 36/36 ingestor tests pass (existing `IngestionServiceIntegrationTest` exercises the public `ingestUploadedFile` API end-to-end unchanged; Testcontainers-dependent, skips in this sandbox for the pre-existing Docker-API-version reason, same as every other Testcontainers suite in this repo).

- [x] **2.4 Ingestion-completed signal → subscriptions fire + caches invalidate.**
  - Done: bridged the previously intra-service-only `DocumentIngestionCompletedEvent` across to `documentation-bot` using the originally-designed Postgres LISTEN/NOTIFY shape. Ingestor's `IngestionEventListener.onIngestionCompleted` now also issues `NOTIFY docai_ingestion_completed, '<documentId>'` (via `JdbcTemplate`, only on `event.success()` — a failed ingestion has nothing new to tell a subscriber about) after syncing webhook status. Payload is just the document id; the bot already has read access to the shared `documents` table for everything else (tenantId/product/version/documentName/status), so there's no need to duplicate that into the notification payload.
  - New bot-side `IngestionCompletedNotifyListener`: holds one dedicated JDBC connection for the app's lifetime (LISTEN's "is listening" state is per-connection, so it can't come from the pooled `DataSource` transiently), issues `LISTEN docai_ingestion_completed`, and blocks in a loop on `PGConnection.getNotifications(30_000)` on a dedicated daemon thread — reconnects with a 5s backoff on any failure. On each notification, looks up the document, and if `COMPLETED`, sets `TenantContext` (this runs on a background thread outside any request, same reasoning as `DocumentationGapService`'s scheduled-job handling from Phase 1.1) and calls `TopicSubscriptionService.notifySubscribersForProduct`.
  - Accept: not verified live end-to-end (no running stack with two live services + Postgres in this environment) — verified by reasoning through both sides of the bridge plus full `mvn test` (bot 122/122, ingestor 36/36, no regressions). A real run would need: subscribe to a topic/product → upload a new version → notification row appears.

## Phase 3 — Chat streaming + stop + error boundary  `[CRITICAL UX — requirement 1]`

**Goal:** Answers stream token-by-token like ChatGPT/Claude/Perplexity; users can stop generation; a render error never white-screens the app.

- [x] **3.1 SSE streaming endpoint.**
  - Done: `POST /api/chat/query/stream` (`ChatController.queryStream`) returns `text/event-stream` via `SseEmitter`. `ChatService.processQueryStream` (`@Async`, runs on the TaskDecorator-wrapped pool so `TenantContext` propagates) mirrors `processQuery`'s full pipeline — session get/create, query analysis, scope resolution, prefs/context — then either streams real tokens (single-hop path) or runs the existing blocking multi-hop pipeline and flushes it as one `token` event (documented scoping decision — see 3.1 note below). Events: `sources` (once, before generation) → any number of `token` → `done` (messageId/confidence/sessionTitle/relatedQuestions/reasoningChain) → `error` on failure. Persistence (message saves, session update, title generation, `ChatQueryRecordedEvent` publish) happens after the stream completes, in the same style as the blocking path, so `AnalyticsService`/`PeopleAlsoAskedService`'s `AFTER_COMMIT` listeners still fire correctly.
  - New supporting pieces: `LLMProvider.stream(...)` (both `OpenAILLMProvider`/`AnthropicLLMProvider`, using Spring AI's `ChatModel.stream(Prompt)` → `Flux<ChatResponse>`, already transitively on the classpath — no new dependency), `LLMRouter.streamChat(...)` (same tenant/key/fallback resolution as `chatWithUsage`, wrapped in `Flux.defer` so a synchronous `TenantContext` failure surfaces as a normal `onError` instead of leaking resilience permits), `AnswerGenerationService.streamAnswer(...)` (hand-rolled bulkhead/circuit-breaker permit acquire/release around the Flux — no `resilience4j-reactor` module is on the classpath, confirmed via dependency:tree, so this isn't the declarative operator style used elsewhere, just the same protection applied manually), and `promptForStreaming`/`emptyStateMessage`/`parseStreamedOutput` (package-private extractions from `generateAnswer` so the streaming path builds/parses the identical prompt without duplicating the confidence-caveat/cross-version logic).
  - **Scoping decision** (disclosed, not hidden): only the common non-multi-hop path streams real tokens. Reworking `MultiHopReasoningService`'s decompose→search→synthesize pipeline for token streaming is separate, larger work; complex queries still get a correct answer, just not a token-by-token one, delivered through the same SSE contract so the frontend never special-cases query complexity.
  - Fixed while wiring this: `RateLimitFilter` matched only the literal string `/api/chat/query` — extended to also match `/api/chat/query/stream` (same per-user bucket, since both cost the same LLM call).
  - Verified: `mvn test` — 122/122 bot tests pass. **Not verified live** (no running stack with real LLM credentials in this environment) — logic was verified by full reasoning through the code path, not by an actual `curl -N` streaming session.
- [x] **3.2 Frontend token-by-token rendering + stop.**
  - Done: `chatService.ts` gained `streamChatMessage(...)` + a hand-rolled SSE-over-`fetch` parser (`parseSSEStream` — `EventSource` can't do POST + `Authorization` headers, so this reads `response.body`'s `ReadableStream` directly and splits on `\n\n`/`event:`/`data:` per the SSE wire format). `ChatPage.handleSendMessage` (`App.tsx`) now streams: creates an `AbortController`, adds a `isTyping` placeholder message, and patches it in place via a new `updateMessageByLocalId` (matches the client-generated `id`, not the backend `messageId`, which isn't known until `done`) as `sources`/`token`/`done` events arrive. Token deltas are coalesced into at most one state update per animation frame (`requestAnimationFrame`) rather than one `setState` per delta.
